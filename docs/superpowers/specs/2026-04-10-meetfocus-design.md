# MeetingFocuser (meetfocus) — Design Spec

## Context

在学术报告和技术讲座中，演讲者经常分享论文之外的实践细节（如数据收集流程、工程经验等），这些内容无法拍照获取，手动记笔记又容易遗漏关键细节。MeetingFocuser 旨在解决这一问题：自动录音→转写→AI 总结→归档到知识库，让参会者专注于听讲和提问。

## Overview

MeetingFocuser 是一个 Python CLI 工具，实时录制会议/报告音频并用本地 Whisper 模型转写为文字，会后通过可配置的 LLM（Claude/OpenAI 等）生成结构化总结，最终以 Markdown 文件写入 Obsidian vault。

## Architecture

采用**录音/转写双进程分离 + 文件中转**架构：

```
┌───────────────────────────────────────────────────────┐
│  CLI Orchestrator  (meetfocus start/stop/summarize)   │
│                                                       │
│  ┌────────────┐    磁盘 .wav 块    ┌──────────────┐  │
│  │  Recording  │──── chunks/ ─────▶│ Transcription │  │
│  │  Process    │                   │ Process       │  │
│  │             │                   │               │  │
│  │ 🎤 Mic     │                   │ faster-whisper│  │
│  │ 🔊 System  │                   │ → transcript  │  │
│  └────────────┘                   └──────────────┘  │
│                                                       │
│  Session Dir: ~/.meetfocus/sessions/<timestamp>/       │
│    ├── chunks/          (audio chunks .wav)            │
│    ├── transcript.txt   (incrementally appended)       │
│    ├── summary.md       (generated by summarize cmd)   │
│    └── meta.json        (session metadata)             │
│                                                       │
│           meetfocus summarize [session_id]             │
│                       │                               │
│                       ▼                               │
│              ┌──────────────┐                         │
│              │  Summarizer   │                         │
│              │  Claude/GPT/  │                         │
│              │  configurable │                         │
│              └──────┬───────┘                         │
│                     │                                 │
│                     ▼                                 │
│              ┌──────────────┐                         │
│              │ Obsidian      │                         │
│              │ Writer        │                         │
│              │ → vault/*.md  │                         │
│              └──────────────┘                         │
└───────────────────────────────────────────────────────┘
```

**设计要点：**
- 录音进程和转写进程独立运行，一方崩溃不影响另一方
- 音频块写入磁盘天然持久化，即使进程异常也不会丢失已录制的音频
- `summarize` 是独立命令，与录音/转写解耦，可以多次运行、使用不同 prompt

## CLI Commands

```
meetfocus start [--system]    # 开始录音 + 实时转写（前台运行，显示实时状态，Ctrl+C 优雅停止）
meetfocus stop                # 停止录音，等待剩余转写完成（仅在 start 以 daemon 模式运行时需要）
meetfocus summarize [id]      # 对某次会议调用 LLM 生成总结 → 写入 Obsidian
meetfocus status              # 查看当前录音/转写状态
meetfocus config              # 查看/编辑配置
meetfocus list                # 列出历史会议
meetfocus show <id>           # 查看某次会议的转写/总结
```

**运行模式**：V1 默认前台运行 — `meetfocus start` 启动后在终端显示录音状态（时长、已转写块数等），用户按 Ctrl+C 优雅停止。这对会议场景足够（笔记本开着，终端在后台）。`--system` 标志切换到系统音频（BlackHole）。

## Components

### 1. Recorder (`recorder.py`)

**职责**：捕获音频，分块写入磁盘。

- 使用 `sounddevice` 库进行音频采集
- 支持两种输入源：
  - **麦克风**：直接录制默认或指定的输入设备
  - **系统音频**：通过 BlackHole 虚拟设备捕获（用户需预先安装 BlackHole 并配置音频路由）
- 音频参数：16kHz 采样率，mono，WAV 格式
- 每 30 秒生成一个音频块文件到 `session_dir/chunks/`
- 作为独立子进程运行（与 transcriber 进程隔离），通过信号接收停止指令

### 2. Transcriber (`transcriber.py`)

**职责**：监听音频块目录，用 Whisper 逐块转写。

- 使用 `faster-whisper`（CTranslate2 后端），比官方 Whisper 快约 4 倍
- 计算后端：CPU 和 CUDA 可用。CTranslate2 不支持 Apple Silicon MPS/Metal — macOS 上使用 CPU + Apple Accelerate 框架，`compute_type="auto"`（在 ARM64 上自动选择 int8）
- Whisper 模型大小可配置（默认 `large-v3`），语言默认 `zh`
- 使用 `watchdog` 监听 chunks 目录，有新 .wav 文件即转写
- 转写结果追加写入 `session_dir/transcript.txt`
- 维护已处理文件列表，避免重复转写
- 独立进程运行，recording 停止后继续处理剩余 chunks

### 3. Summarizer (`summarizer.py`)

**职责**：读取完整转写文本，调用 LLM 生成结构化总结。

- **Provider 抽象层**：统一接口支持多种 LLM
  - Claude（通过 `anthropic` SDK）
  - OpenAI（通过 `openai` SDK）
  - 可扩展其他 provider
- Provider 和模型通过配置文件选择
- **Prompt 设计**：指导 LLM 生成以下结构：
  - 摘要：2-3 句话概括
  - 详细内容：根据演讲者的逻辑结构自动分块（按论文/项目 或 按专题），保留充分细节，不过度压缩
  - Q&A：提问环节的问答对
  - 行动项：可执行的 TODO 列表
- **详细内容的输出原则**：
  - 跟随演讲者的原始组织方式（按时间线/项目/专题等）
  - 保留具体数字、方法名、工具名、论文名等细节
  - 特别标注论文/文章中未公开的信息
  - 保留足够上下文，使后续 AI agent 能基于笔记进行问答
- 对于长转写文本（>100K tokens），使用支持大上下文的模型

### 4. Writer (`writer.py`)

**职责**：将总结写入 Obsidian vault。

- 生成包含 Obsidian frontmatter 的 Markdown 文件：
  ```yaml
  ---
  date: 2026-04-10
  speaker: 演讲者名称
  topic: 主题
  duration: 45min
  tags: [meeting, ...]
  ---
  ```
- 文件名格式可配置（默认 `{date}-{title}.md`）
- 写入路径：`{vault_path}/{folder}/` （folder 可配置，默认 `Meetings`）
- 可选同时保存转写文本（作为同目录下的 `{date}-{title}-transcript.md`）

### 5. Session Manager (`session.py`)

**职责**：管理会议会话的生命周期。

- 每次 `start` 创建新的 session 目录：`~/.meetfocus/sessions/{YYYY-MM-DD_HHMM}/`
- `meta.json` 记录会话元数据：开始时间、结束时间、音频源、状态等
- 提供 session 列表、查询、清理接口
- 管理产物保留策略（根据配置决定是否保留音频/转写/总结）

### 6. Config (`config.py`)

**职责**：配置文件管理。

配置文件路径：`~/.meetfocus/config.toml`

```toml
[audio]
device = "default"        # 具体设备名或 "default"
source = "mic"            # "mic" | "system"
chunk_duration = 30       # 每块音频秒数
sample_rate = 16000

[whisper]
model = "large-v3"        # tiny/base/small/medium/large-v3
language = "zh"
device = "auto"           # "cpu" | "cuda" | "auto" (no MPS — CTranslate2 不支持 Metal)

[llm]
provider = "claude"       # "claude" | "openai"
model = "claude-sonnet-4-6"
api_key_env = "ANTHROPIC_API_KEY"

[output]
obsidian_vault = "~/Obsidian/MyVault"
folder = "Meetings"
keep_transcript = true
keep_audio = false
filename_format = "{date}-{title}"

[sessions]
dir = "~/.meetfocus/sessions"
```

### 7. CLI Entry Point (`cli.py`)

**职责**：CLI 命令定义和进程编排。

- 使用 `typer` 框架
- `start`：创建 session → 启动 recorder 子进程 → 启动 transcriber 子进程 → 前台显示状态（时长、chunk 数、转写进度）
- Ctrl+C / `stop`：发送停止信号给 recorder → 等待 transcriber 处理完剩余 chunks → 显示 session 完成信息
- `summarize`：加载 transcript → 调用 summarizer → 调用 writer
- 进程管理：使用 PID 文件跟踪运行中的进程


## Error Handling

- **录音崩溃**：已写入磁盘的音频块不受影响，transcriber 可继续处理已有 chunks
- **转写崩溃**：音频块持久化在磁盘，可重启 transcriber 从断点继续
- **LLM API 失败**：transcript 已保存，可多次重试 `summarize` 命令
- **磁盘空间**：启动时检查可用空间，录音中监控空间告警
- **BlackHole 未安装**：使用 `--system` 时检测 BlackHole 设备是否存在，提示安装指引

## Testing

- **单元测试**：各组件独立测试（mock 音频输入/LLM API）
- **集成测试**：用预录音频文件测试完整流程 start→stop→summarize
- **手动验证**：实际录制一段短音频，验证转写质量和总结格式
